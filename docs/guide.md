# 에이전트 참고 문서 운영 가이드

이 문서는 사람이 `AGENTS.md`, `CLAUDE.md`, `docs/agent/` 문서 체계를 관리할 때 참고하기 위한 가이드다. 에이전트가 기본 작업 규칙으로 읽지 않도록 `AGENTS.md`와 `CLAUDE.md` 라우팅에서 제외한다.

## 목적

에이전트 문서는 길수록 좋은 문서가 아니다. 목표는 에이전트가 작업할 때 자주 틀리는 판단 지점을 짧고 명확한 규칙으로 제공하는 것이다.

좋은 에이전트 문서는 다음을 만족한다.

- 기본 컨텍스트 비용이 작다.
- 작업 종류별로 필요한 문서만 읽게 한다.
- 원칙만 말하지 않고 적용 조건과 예외를 함께 제공한다.
- 예시는 필요한 경우에만 읽는 선택 문서로 분리한다.
- 사람이 원하는 개발 철학을 에이전트가 실행 가능한 판단 기준으로 바꾼다.

## 문서 체계

`AGENTS.md`와 `CLAUDE.md`는 라우터다. 실제 규칙을 길게 적지 않고, 어떤 작업에서 어떤 문서를 읽어야 하는지만 알려준다.

`docs/agent/`는 에이전트가 실제로 읽는 규칙 문서 모음이다.

- `coding-style.md`: React/TypeScript 코드 작성과 컴포넌트 구성 규칙
- `ui-style.md`: UI, 컴포넌트, 화면 스타일 규칙
- `testing-style.md`: 테스트 작성 원칙과 계층별 테스트 선택 기준
- `commit-convention.md`: 커밋 메시지와 브랜치 전략 기준
- `documentation-style.md`: `AGENTS.md`, `CLAUDE.md`, `docs/agent/` 문서를 수정할 때의 문서 작성 기준

사람만 참고할 설명, 배경 철학, 긴 rationale은 `docs/` 아래에 두되 `AGENTS.md`와 `CLAUDE.md` 라우팅에는 연결하지 않는다.

## 기본 원칙

규칙은 길게 설명하기보다 타이브레이커처럼 작성한다.

나쁜 예:

```md
방어 로직을 과하게 넣지 않는다.
```

좋은 예:

```md
서버 응답 DTO 등 외부 입력은 `unknown`으로 받고 좁힌다.
다만 내부에서 만든 값처럼 타입상 null이 들어올 수 없는 경로에는 중복 방어 코드를 추가하지 않는다.
```

두 번째가 더 좋은 이유는 에이전트가 “과하게”를 해석하지 않아도 되고, 적용 조건과 예외가 함께 있기 때문이다.

## 규칙 작성 형식

새 규칙을 추가할 때는 가능한 한 아래 순서로 쓴다.

1. 의도: 왜 이 규칙이 필요한가
2. 선호 방향: 기본적으로 어떤 선택을 할 것인가
3. 적용 조건: 어떤 상황에서 이 규칙을 적용할 것인가
4. 예외: 언제 반대 선택이 가능한가
5. 리뷰 질문: 작업 전 스스로 무엇을 물어볼 것인가

모든 규칙에 이 형식을 강제할 필요는 없다. 하지만 에이전트가 판단을 자주 틀리는 영역이라면 이 구조가 가장 안정적이다.

## 문서 배치 기준

새 규칙을 어디에 둘지는 다음 기준으로 정한다.

- 코드 작성 중 항상 필요하면 `coding-style.md`
- UI나 화면 작업 중 항상 필요하면 `ui-style.md`
- 테스트 작성 중 항상 필요하면 `testing-style.md`
- 커밋 메시지나 브랜치 기준이면 `commit-convention.md`
- 문서 작성 방식 자체에 대한 규칙이면 `documentation-style.md`
- 사람이 이해해야 할 배경 설명이면 `docs/`

짧은 타이브레이커 하나로 충분하면 새 문서를 만들지 않는다. 이미 작업 중 읽는 문서에 5~15줄 정도로 추가하는 편이 낫다.

긴 예시나 배경 설명이 필요하면 별도 문서로 분리하고, `AGENTS.md`와 `CLAUDE.md` 라우팅에는 필요한 작업에서만 읽도록 연결한다.

## 컨텍스트 비용 줄이는 방법

에이전트 문서를 관리할 때는 다음 순서로 생각한다.

1. 이 규칙이 에이전트가 실제로 자주 틀리는 판단인가?
2. 기존 코드 구조나 타입, 린트로 강제할 수 있는가?
3. 이미 읽는 문서에 짧은 타이브레이커로 넣을 수 있는가?
4. 예시가 꼭 필요한가?
5. 예시가 길다면 선택 문서로 분리할 수 있는가?

문서보다 코드 구조가 먼저다. 문서는 코드 구조로 강제하기 어렵거나, 여러 선택지 사이에서 에이전트가 흔들리는 지점을 보정하는 도구로 사용한다.

## 업데이트 체크리스트

`AGENTS.md`, `CLAUDE.md`, `docs/agent/`를 수정할 때는 다음을 확인한다.

- `AGENTS.md`와 `CLAUDE.md`가 라우터 역할만 유지하는가?
- 새 규칙이 실제 작업에서 필요한 문서에 배치되었는가?
- 원칙만 있고 적용 조건이나 예외가 빠져 있지 않은가?
- 오래된 코드 예시가 현재 API와 어긋나지 않는가?
- 긴 예시를 기본 문서에 넣어 컨텍스트 비용을 늘리지 않았는가?
- 사람만 읽을 배경 설명을 에이전트 기본 참고 문서에 넣지 않았는가?
- `.claude/settings*.json` 같은 특정 도구 설정을 공통 규칙으로 오해하게 만들지 않았는가?

## 운영 팁

에이전트가 같은 실수를 반복하면 바로 긴 철학 문서를 추가하지 말고, 먼저 기존 규칙 문서에 짧은 판단 기준을 추가한다.

예를 들어 “방어 로직을 너무 많이 넣는다”는 문제가 반복되면 “방어 로직을 줄여라”가 아니라 “정상 코드 경로에서 값이 들어올 수 있는가, 아니면 타입상 불가능한가?” 같은 리뷰 질문을 추가한다.

반대로 사람이 팀 철학을 정리하고 싶지만 에이전트가 매번 읽을 필요는 없다면 이 문서처럼 `docs/` 아래에 두고 `AGENTS.md`와 `CLAUDE.md` 라우팅에는 연결하지 않는다.
